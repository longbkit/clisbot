Hermes Agent Self-Evolution Architecture

Technical map of the two mechanisms behind the "self-improving agent" claim: runtime learning inside Hermes Agent, and offline eval-driven artifact evolution in the companion self-evolution repo.
Runtime Hermes Offline evolution pipeline Gates and persistence Caveat or risk

1. Live Agent Learning Loop

User work through CLI, gateway, ACP, cron

AIAgent.run_conversation() handles prompt assembly, model calls, tool dispatch, retries, and session persistence.

↓

Tool use and conversation traces

Tool iterations, user turns, assistant outputs, tool results, and session metadata become the evidence base for later review.

↓

Memory writes

Durable facts go to MEMORY.md, USER.md, and optional memory providers such as Honcho.

Skill writes

Reusable procedures go to ~/.hermes/skills/ through skill_manage.

↓

Background review fork

After threshold triggers, a quiet forked agent reviews the transcript. Runtime whitelist allows memory and skill tools only.

Curator

Tracks agent-created skills, marks stale entries, archives recoverably, and can patch or consolidate skill sprawl.

2. Offline Self-Evolution Pipeline

Select optimization target

Current implemented path targets SKILL.md. Planned targets include tool descriptions, prompt sections, and code.

↓

Build eval dataset

Synthetic generation, golden JSONL, or mined sessions from Claude Code, Copilot, and Hermes history.

↓

Wrap artifact as DSPy module

For skills, the body text becomes an optimizable instruction parameter: skill_instructions + task_input -> output.

↓

GEPA / MIPROv2 optimization

Mutates text candidates, evaluates them, and uses feedback to search for better variants without GPU training.

↓

Holdout comparison and constraints

Compare baseline vs evolved candidate. Enforce size, growth, structure, tests, benchmarks, and semantic preservation where implemented.

↓

Branch and PR

Intended deployment is a reviewable diff with metrics. Human review remains in the merge path.

3. Persistence Surfaces

Semantic memory

MEMORY.md, USER.md, Honcho and other memory providers build durable user/environment context.

Procedural memory

Skills capture workflows, commands, pitfalls, verification steps, and recurring task procedures.

Episodic memory

SQLite sessions with FTS5 search power session_search and later mining for eval cases.

Trajectory memory

ShareGPT-style JSONL from agent/trajectory.py and batch_runner.py supports eval and training workflows.

What it is not

Not live model-weight self-training. Not arbitrary self-modifying code in the active conversation. Not auto-merge.

Main maturity caveat

Skill evolution is the only implemented phase in the companion repo. Tool, prompt, code, and continuous evolution are roadmap-level.

Implemented Phase 1: Skills DSPy wrapper, dataset builders, importers, fitness, constraints, evolve_skill entrypoint.
Planned Phase 2: Tool descriptions Optimize tool schema descriptions and evaluate tool selection accuracy.
Planned Phase 3: System prompt Optimize prompt sections with broad behavioral and benchmark gates.
Planned Phase 4: Code Use Darwinian Evolver externally for tool implementation code under strict tests.
Planned Phase 5: Continuous loop Monitor weak areas, schedule optimization, generate PRs, keep human merge gate.

Create Skill

Live runtime writes procedural memory after complex successful tasks, recovered errors, user corrections, or explicit "remember this procedure" requests.

Name And Shape

Names are lowercase filesystem-safe ids, max 64 chars. Good scale names use domain + workflow, such as github-pr-review.

Scale Failure Mode

Too many narrow or overlapping skills can make the index noisy, trigger wrong skill loads, and turn procedural memory into operational drag.

Evolve Skill

Offline pipeline wraps SKILL.md as a DSPy module, builds eval data, runs GEPA/MIPROv2, compares holdout scores, then saves a reviewable diff.

DSPy Wrapper

The skill body becomes an instruction parameter. Eval calls look like skill_instructions + task_input -> output; frontmatter is preserved when reassembled.

Optimizer Defaults

Current CLI defaults: 10 GEPA steps, synthetic dataset of 20 examples, split about 10/5/5 train/val/holdout.

Hard Gates

Skill max size is 15,000 chars and growth max is 20%. Tool descriptions are capped at 500 chars, params at 200.

Current Weakness

Main scoring is keyword overlap, not full agent execution. The exposed test flag exists, but the inspected main flow does not call the full test gate.

Context Still Decays

Hermes can still get worse when tool output, loaded skills, and recalled memory crowd the prompt. It manages context debt; it does not abolish it.

20% Pick: Skills

Use skills for repeated procedures, recovered errors, and user-corrected workflows. Patch existing skills before creating new ones.

20% Pick: Recall

Use session search and memory as retrieval aids, then verify source files or commands before project-status claims and mutations.

20% Pick: Governance

Keep the default skill set small, use domain + workflow names, run curator dry-runs, and archive stale skills recoverably.

20% Pick: Evals

Turn repeated failures into trajectory and eval cases, then evolve selected skills under holdout comparison instead of trusting vibes.

One-sentence read: Hermes has a real bounded learning loop for memory and skills today, but it still needs strict context and skill governance; otherwise it can reproduce the same long-context degradation pattern with more moving parts.
