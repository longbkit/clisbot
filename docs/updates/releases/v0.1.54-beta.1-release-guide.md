# clisbot v0.1.54-beta.1 Release Guide

Use this guide when testing the `0.1.54-beta.1` beta from `0.1.53`.

## What Changed

`0.1.54-beta.1` is the first API channel hardening beta after `v0.1.53`.

The main things to notice:

- API bot ingress, result polling, and `actions.message.send` now have a clearer
  Chatwoot/Jira operator path.
- API result persistence is safer when the HTTP listener and one-shot
  `clisbot message send --channel api` write result records at the same time.
- API listener default port is now `6868`.
- Explicit `--reply-to <eventId>` now uses that event's reply metadata for API
  provider actions.
- Global runner admission/backpressure for high-burst API traffic is documented
  as planned work, not implemented in this beta.

## Update

```bash
npm install -g clisbot@0.1.54-beta.1
clisbot restart
clisbot status
```

Or use the beta dist-tag:

```bash
npm install -g clisbot@beta
clisbot restart
clisbot status
```

## Migration

Manual action: none.

There is no new schema migration in this beta. Existing explicit API listener
configuration wins; otherwise the default listener port is `6868`.

## What To Check After Restart

```bash
clisbot --version
clisbot status
```

Expected version: `0.1.54-beta.1`.

For API channel validation:

- Ingress should accept configured API events and return the expected result URL.
- Polling the event result should show progress/final output.
- `clisbot message send --channel api --reply-to <eventId>` should update the
  matching result record.
- If `actions.message.send` is configured, provider delivery should use the
  reply metadata from the target event.

## Known Risks

- There is still no global runner pool or global concurrent runner-start limit.
  Bursts across many API conversations can still start many runner sessions.
- API media/file delivery is not part of this beta.
