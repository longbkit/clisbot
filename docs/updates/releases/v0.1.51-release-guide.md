# clisbot v0.1.51 Release Guide

Date: 2026-05-04

## Read This If

- You saw fresh runner startup fail too quickly on the first prompt.
- You run Codex, Claude, or Gemini through slower machines, heavier workspaces, or colder shells.
- You are updating from an older install such as `0.1.43` and want to know whether this release changes the migration story.

## TL;DR

`v0.1.51` is a small stability release. The main change is that clisbot now gives fresh interactive runners up to 60 seconds to become ready by default instead of timing out much earlier.

The package version moves to `0.1.51`, but the config schema stays at `0.1.50`. That means:

- no new manual migration
- existing explicit startup overrides still win
- old installs before `0.1.50` still get the same automatic backup and config rewrite path as before

## What Changed

- Codex, Claude, and Gemini now inherit a 60-second default startup window.
- The shared schema-level fallback startup default now also matches that 60-second window.
- The stable update docs now point to `0.1.51` as the current package target while keeping the config-shape story truthful: schema remains `0.1.50`.

## What Did Not Change

- No new config schema version.
- No new manual migration runbook.
- No change to how agent-level or CLI-family `startupDelayMs` overrides win over defaults.
- No change to first-prompt submit behavior once the runner is actually ready; clisbot still submits as soon as readiness is confirmed.

## Update / Rollout

- Install or run the `0.1.51` package.
- Restart clisbot.
- Run `clisbot status`.
- If startup slowness was the problem, verify one fresh routed prompt on the affected surface.

## Good First Checks

- One fresh prompt in the slowest Codex workspace you have.
- One fresh prompt on Claude or Gemini if those were timing out before.
- One `clisbot status` check to confirm the runtime is healthy after restart.
