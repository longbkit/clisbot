# clisbot v0.1.52 Release Guide

Date: 2026-05-06

## Read This If

- You were still confused about whether shared-route setup needs both `routes add ...` and `routes set-agent ...`.
- You upgraded to `v0.1.51` but some installs still seemed to behave like they had older short startup timeouts.
- You want the shortest truthful summary of what changed in the current stable release.

## TL;DR

`v0.1.52` is a small truthfulness release.

Two things matter:

- shared-route guidance now says more clearly that `routes add ...` uses the agent currently assigned to that bot by default
- stale short `startupDelayMs` overrides below 30 seconds are now pruned from more current-schema config paths, so the 60-second startup default introduced in `v0.1.51` can actually take effect

The package version moves to `0.1.52`, but the config schema still stays at `0.1.50`.

## What Changed

- Slack and Telegram route guidance is clearer in chat surfaces and docs.
- Current-schema config cleanup now removes more stale short startup-delay pins.
- Queue workflow docs now state thread/topic targeting more explicitly for Slack and Telegram examples.

## What Did Not Change

- No new config schema version.
- No new manual migration runbook.
- No change to the rule that explicit non-stale startup-delay overrides still win over defaults.
- No change to the direct-upgrade path from older installs.

## Update / Rollout

- Install or run the `0.1.52` package.
- Restart clisbot.
- Run `clisbot status`.
- If you use shared surfaces, verify one fresh Slack or Telegram route-onboarding flow.
- If runner startup slowness was an issue, verify one fresh prompt in the slowest workspace you care about.

## Good First Checks

- One unrouted Slack or Telegram shared-surface prompt, to confirm the guidance now matches the intended route mental model.
- One fresh prompt in a slower Codex, Claude, or Gemini workspace.
- One `clisbot status` check after restart.
