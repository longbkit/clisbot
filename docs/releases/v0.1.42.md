# clisbot v0.1.42

Released: 2026-04-20

`v0.1.42` makes runtime recovery more persistent, adds clearer scoped follow-up controls, and tightens routed-command guidance for Slack and Telegram.

## In One Line

`clisbot` now keeps retrying more truthfully when the runtime or Telegram polling hits known transient failure modes, and routed conversations have clearer follow-up controls, transcript views, and command guidance.

## Highlights

### 1. Runtime restart backoff is smoother and no longer stops at the old hard ceiling

- the default runtime monitor ladder now ramps through 1 minute, 3 minutes, 5 minutes, 10 minutes, 15 minutes, and finally 30 minutes
- once the final stage is reached, the runtime keeps retrying at that delay instead of permanently giving up
- older configs that still match the legacy default ladder are normalized in memory to the smoother ladder without rewriting the operator file
- runtime status and tests now cover the new backoff behavior directly

### 2. Telegram polling conflicts recover inside the runtime instead of forcing a hard stop

- when another process is already using the same bot token for `getUpdates`, the Telegram channel now stays alive and retries automatically with backoff
- channel lifecycle can surface the conflict as failed while recovery is in progress, then flip back to active after polling resumes
- delayed owner alerts can now be triggered for persistent runtime-channel failures

### 3. Message-tool prompts and CLI help are clearer

- `message send` now prefers `--file` as the generic attachment flag, with `--media` kept as a compatibility alias
- agent prompt guidance now keeps the readable-markdown instruction while adding short, channel-specific message-length guidance
- prompt wording no longer implies that the model should emit raw destination-native payloads itself

### 4. Follow-up controls are more practical inside routed conversations

- `/mention` is now the fast shorthand for `/followup mention-only`
- `/mention channel` persists mention-only as the default for the current Slack channel, Telegram group, or DM container and applies it to the current conversation immediately
- `/mention all` persists mention-only as the default for the current bot and applies it to the current conversation immediately
- `/followup mention-only channel` and `/followup mention-only all` are now first-class command forms instead of implicit operator knowledge
- routed guidance, Telegram command registration, README, and slash-command docs now surface these shortcuts more directly

### 5. Transcript and bot-ingress behavior are easier to understand

- `/transcript` now returns a compact recent session snapshot by default, which is easier to scan in chat
- `/transcript full` still exposes the longer pane snapshot when the operator needs the full tmux context
- Slack and Telegram guidance now point users toward `/mention` as part of the normal routed-command flow
- `allowBots` remains safely defaulted to `false`, and the release now adds explicit regression coverage that routed bot-originated messages are dropped unless that config is intentionally enabled
