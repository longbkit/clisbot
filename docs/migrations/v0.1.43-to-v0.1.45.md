# clisbot v0.1.43 to v0.1.45 Migration

## Who Needs This

Use this runbook when upgrading an existing `0.1.43` clisbot install to `0.1.45`.

It matters most for installs that use:

- Slack or Telegram shared groups
- Telegram topics
- route allowlists or blocklists
- scheduled loops
- route, bot, agent, or app timezone settings
- Codex, Gemini, or customized runner startup defaults

## What Changes

- Config schema moves to `0.1.45`.
- The first read of a released `0.1.43` config writes a backup under `~/.clisbot/backups/`.
- Legacy shared route keys are migrated into `directMessages` and `groups`.
- Exact shared group and topic routes without explicit policy inherit from `groups["*"].policy`.
- Legacy timezone defaults move into `app.timezone`.
- Release-owned runner startup defaults are pruned from persisted config so the runtime can inherit the current release behavior.

## Preflight

Check the current runtime and config path:

```bash
clisbot status
```

Optional but useful before high-risk upgrades:

```bash
clisbot routes list --channel slack --json
clisbot routes list --channel telegram --json
clisbot loops list
```

## Upgrade

Install the `0.1.45` package, then restart or start the runtime normally:

```bash
npm install -g clisbot@0.1.45
clisbot restart
```

If using a beta package during pre-release validation, install the exact beta version instead:

```bash
npm install -g clisbot@0.1.45-beta.4
clisbot restart
```

The migration runs when clisbot reads the old config.

## Verify

Confirm runtime health:

```bash
clisbot status
```

Confirm shared routes look intentional:

```bash
clisbot routes list --channel telegram --json
clisbot routes list --channel slack --json
```

Look specifically for:

- Telegram topic routes still using the expected `topic:<chatId>:<topicId>` identity.
- `group:*` carrying the intended default sender policy.
- Exact shared routes omitting `policy` when they should inherit from `group:*`.
- No important route unexpectedly set to `policy: disabled`.
- Loops showing the expected next run and timezone.

## Rollback

Use the backup created before migration:

```text
~/.clisbot/backups/clisbot.json.0.1.43.<timestamp>
```

Stop clisbot before restoring the backup, then install or run the older runtime that understands the restored schema.

Dedicated downgrade and restore CLI UX is tracked separately.
